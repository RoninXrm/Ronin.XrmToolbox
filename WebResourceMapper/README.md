# Web Resource Mapper (XrmToolBox Plugin)

This is an **XrmToolBox plugin** for managing Dataverse web resources with a local folder workflow.

If you’ve ever bounced between web resources in Dataverse and files on disk and thought _"there has to be a cleaner way"_, this is for you.

This project was inspired by [**D365 Developer Extensions / D365 Developer Tools**](https://github.com/tsharp/D365DeveloperExtensions) and its web resource workflow in Visual Studio. I wanted something that gave me that same kind of experience while working in **VS Code** instead of being tied to full Visual Studio.

---

## What it does

- Loads **unmanaged solutions**
- Loads solution web resources via `solutioncomponent` (component type `61`)
- Lets you map web resources to local files
- Supports a root-folder-based workflow with **relative paths**
- Download selected web resources to disk
- Upload new web resources (with a naming dialog)
- Update selected web resources from local files
- Publish updated web resources automatically
- Bulk operations using DataGridView checkboxes
- Save mappings in plugin settings (grouped by connection + solution)

---

## UI at a glance

- Solution picker
- Root folder picker
- Web resources grid with mapping column
- Actions:
  - Load Web Resources
  - Download Selected
  - Upload New
  - Update Selected
  - Reset + Auto Map

---

## Mapping behavior highlights

- Mappings are persisted per **connection + solution**
- Paths are stored as **relative to root folder**
- Prefix-folder download logic supports publisher prefixes like `new_`
- "No" answers for prefix-folder creation can be remembered per solution

---

## Tech stack

- C# / WinForms
- .NET Framework 4.8
- XrmToolBox Plugin SDK (`PluginControlBase`)
- Dataverse SDK (`IOrganizationService`, `QueryExpression`)

---

## Local development

1. Open solution in Visual Studio
2. Build `Debug`
3. Run/debug with XrmToolBox plugin workflow

> The project post-build step copies the plugin assembly to a local `Plugins` folder in `bin\Debug`.

---

## Why this exists

I really liked the web resource workflow in D365 Developer Extensions, especially the practical day-to-day loop of mapping, downloading, updating, and publishing. This plugin exists to bring that same style of workflow into my XrmToolBox + VS Code setup, with solution-aware mappings and local-folder control.

## Approach

This entire plugin was developed through **vibe coding with GitHub Copilot** 🤖✨

### Prompt

The following intial prompt was given to MS Copilot.

> *I want to vibe code an XrmToolbox plugin using Visual Studio and Github Copilot. Here is my starting prompt. Please review and suggest edits that may help Github Copilot get off to a good start:*
>
> *I want to create an XrmToolbox plugin to manage solution based WebResources in a Dataverse environment. Developer documentation for XrmToolbox may be found here: https://www.xrmtoolbox.com/documentation/for-developers/*
> 
> *It should take inspiration from D365 Developer Extensions, Web resource deployer for functionality. https://github.com/tsharp/D365DeveloperExtensions*
> 
> *The plugin should have the following functionality:*
> - *A solution drop-down that only shows unmanaged solutions.*
> - *When selected, a table lists the WebResources in that solution.*
> - *A Directory picker to select the local filesystem root folder for the WebResources.*
> - *Each row in the WebResources table should have the ability to select the local file to map to.*
> - *There should be the ability to download a WebResource from Dataverse to the local file system. This should auto map.*
> - *There should be the ability to create a new WebResource in Dataverse from a file in the local filesystem. It should be added to the selected solution.*
> - *There should be the ability to update a WebResource from a mapped file in the local filesystem.*
> - *There should be checkboxes on the rows so I can update multiple WebResources at once.*

MS Copilot gave me a much larger and more structured prompt which I gave to Github Copilot Agent on a new XrmToolbox Plugin project. The agent churned away and actually gave me a working plugin on first attempt. Functionality was then increased and improved through a number of conversational prompts with the Agent.

No lines of code were written by a human. Even this README (apart from this section) was generated by the Agent.

> **NOTE:** Something I found interesting and will consider moving forward. I had reason to leave the project and return the next day to continue adding functionality. When I did, I did so through a new Agent Chat session. After one prompt, there was carnage in the plugin. It didn't deal with it very well at all. I reverted those changes, trashed that Chat session and resumed the previous Chat from History. The output from subsequent prompts were much more sensible and in line with previous output.
